# 03_01 个人信息保护（PIPL）执行指引

## 1. 角色界定（双角色模型，审计 A4-P1-6）

平台方的合规角色**因场景而异**，不能一律套「受托处理方」：

| 场景 | 个人信息处理者 | 平台方角色 |
|------|----------------|------------|
| 企业名片业务（员工名片、访客访问、客户映射） | 企业租户 | **受托处理方** |
| `growth_leads` 引流留资（"我也想要名片") | **平台方** | **独立个人信息处理者** |

- 企业名片业务：企业为处理者，平台为受托处理方，协议中明确权责与数据处理范围。
- `growth_leads` 是平台自己的获客行为，非企业委托，平台为独立处理者：**留资弹窗单独告知处理者是平台方**，不得沿用企业租户的隐私告知；单独同意。
- 处理的个人信息：员工手机号/邮箱/微信号，访客 openid/unionid、行为埋点，客户 external_userid、留资信息。

## 2. 告知与同意（§26.1）

- 首次收集前弹**个人信息处理告知**：处理者、目的、种类、留存期限、对外提供、拒绝影响。
- **敏感个人信息**（手机号等）单独、明确同意，不默认勾选。
- 采集时机对齐小程序：见 [`../01-specs/01_03_Miniprogram_Guide.md`](../01-specs/01_03_Miniprogram_Guide.md) §5（getPhoneNumber、头像昵称填写、留资）。

## 3. 数据主体权利（§26.2）

提供查询 / 更正 / 删除 / 导出入口（员工与访客）：

- 删除：依赖软删除 `deleted_at`（§15.2）+ 定期物理清理 / 匿名化。
- 员工离职、企业取消授权后按时限清理或匿名化对应个人信息（联动 [`01_01_Wecom_Integration.md`](../01-specs/01_01_Wecom_Integration.md) §7/§9）。
- 导出：结构化导出该主体数据。

## 4. 留存与最小化（§26.3）

| 数据 | 留存 | 处理 |
|------|------|------|
| 访问/动作埋点明细 | 定义 N 个月 | 到期聚合后清明细 |
| 客户映射/归属 | 关系存续期 | 删除权触发匿名化 |
| 留资 growth_leads | 转化周期内 | 到期清理 |

只收集名片分发所必需字段；跨租户严禁画像（§4.3/§16）。

## 5. 跨境与第三方 SDK（§26.4）

- 使用境外云/SDK 需评估数据出境合规。
- 维护**第三方 SDK 清单**（名称、收集的个人信息、用途），在隐私政策披露。（清单文件：`03-compliance/03_02_SDK_Inventory.md`，待建。）

## 6. 安全联动（§17）

- PII 加密：KMS 信封 + 可检索 `*_hash`（§15.2/§17.1）。
- 日志脱敏：手机号/邮箱/微信号/access_token/permanent_code/external_userid 全量值不明文（§17.4）。
- 展示开关：手机号默认不展示；企业规则为硬边界（§11.3）。

## 7. 上线合规验收清单（§26.5 / §31.1）

- [ ] 隐私政策 / 用户协议已上线，小程序内可查看。
- [ ] 敏感信息单独同意弹窗。
- [ ] 数据主体删除 / 导出功能可用。
- [ ] 留存期限策略与自动清理任务落地。
- [ ] 第三方 SDK 清单与数据出境登记完成。

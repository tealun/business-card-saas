# 企业微信服务商商业化开发计划

状态：Ready for M0  
计划日期：2026年7月16日  
规格事实源：[`../01-specs/01_07_Commercialization_And_Quota_Spec.md`](../01-specs/01_07_Commercialization_And_Quota_Spec.md)

## 1. 规划契约

- **Done definition**：形成企业微信订单 → 企业订阅 → 共享展示额度 → 公开名片原子扣量 → 管理端余额 → 企业微信通知 → 退款/到期处置的可实施计划，并明确数据库、接口、安全、测试、运维和合规门槛。
- **Validation method**：M0 保存官方文档和脱敏真实报文；M1 使用企业微信测试企业完成一笔测试订单的垂直闭环；每阶段运行迁移校验、后端 lint/typecheck/test/build，并执行跨租户和并发扣量测试。
- **Fail condition**：无法取得订单接口权限、无法把订单可靠关联到授权企业和版本、无法验证回调签名/幂等，或平台不允许固定年费版本附带展示额度时，生产收费路径停止，不能用人工假订单替代审核证据。

## 2. 当前基线

现有可复用能力：

- NestJS + Fastify 后端、PostgreSQL、前向 SQL 迁移；
- 企业微信 suite ticket、suite token、授权、企业 token、指令/数据回调验签解密和事件重试；
- `tenant_id` 上下文、RLS、企业管理员 RBAC；
- 公开名片目录、访问记录、服务端签名匿名访客和 visit token；
- 企业成员名片、管理后台、指标和生产部署流程。

当前缺口：

- 没有订单、订阅、套餐、额度账户和不可变账本；
- 没有企业微信付费订单事件处理；
- 公开名片在计量前已经可以读取，无法执行额度准入；
- 没有按企业展示余额、购买入口、订单和消费明细；
- 没有应用通知 outbox、定时到期提醒和模板 ID 管理；
- 上线材料仍存在“完全免费版”旧口径，收费前必须统一修订；
- 小程序仍存在演示名片降级和“即将上线”入口，不满足“授权即可使用”审核体验。

## 3. 里程碑总览

| 阶段 | 目标 | 生产收费状态 |
|---|---|---|
| M0 | 证明企业微信订单、版本、回调和通知的真实协议 | 禁止 |
| M1 | 一笔测试订单完成订阅入账和一次真实展示扣量 | 禁止，仅测试企业 |
| M2 | 完成三版本、管理端、通知、退款/到期和运营闭环 | 灰度 |
| M3 | 完成压测、审计、客服与财务对账后正式开放 | 允许 |

- Verdict：M0 通过后才是 `Ready for M1 walking skeleton`。

## 5. M1：订单到扣量的垂直切片

### M1.1 数据库迁移

- [ ] 新增 `migrate_v1_10.sql`：套餐、外部版本映射、企业订阅、订单、额度批次、额度账本、展示去重、通知 outbox/delivery。
- [ ] 同步 `database/schema.sql` 和 `docs/00-core/00_02_Database_Schema.md`。
- [ ] 为所有企业表添加 RLS、外键、唯一约束、金额/额度 CHECK 和常用索引。
- [ ] 增加 idempotent seed，将四个套餐写入 `commercial_plans`，但外部版本 ID 由环境配置或受审计的管理入口维护。
- [ ] 编写安全校验 SQL：余额非负、账本和余额一致、订单唯一、跨租户不可见。

迁移执行顺序：

1. 本地一次性 PostgreSQL 测试库执行全新 `schema.sql`；
2. 从当前生产结构执行 `migrate_v1_10.sql`；
3. 执行 grant/RLS 校验和非破坏性一致性查询；
4. 备份生产库后再部署迁移；
5. 迁移先于新代码发布，代码在商业化功能开关关闭时兼容空表。

### M1.2 套餐与订单模块

- [ ] 建立内部 plan code 与外部版本 ID 的显式映射。
- [ ] 实现订单回调验证、解密、幂等持久化和状态机。
- [ ] 测试订单首次成功入账时创建 subscription 和 quota grant。
- [ ] 重复、并发、乱序回调不重复增加额度。
- [ ] 未知版本、金额不符、企业未授权进入 `manual_review`，不自动开通。

### M1.3 展示准入和原子扣量

- [ ] 新增 `POST /public/cards/{public_id}/display-session`。
- [ ] 在单事务内完成企业/名片状态校验、去重锁、额度扣减、账本、访问记录和 visit token。
- [ ] 同一访客同一卡片30分钟内重复打开不重复扣量。
- [ ] 不同访客、不同卡片正确扣量；并发请求不能超扣或出现负数。
- [ ] 员工/管理员预览必须通过服务端认证和所属企业校验后免扣量。
- [ ] 额度用尽、订阅到期和企业取消授权均返回稳定错误码，且不泄露完整名片。
- [ ] 小程序公开页改用新接口；旧公开读取接口撤销完整数据访问或仅保留受认证兼容路径。

### M1.4 最小管理端

- [ ] 企业管理员可查看当前版本、起止时间、总额、已用和剩余。
- [ ] 平台管理员可查看测试订单、内部 plan 与外部版本映射。
- [ ] 功能开关默认关闭，仅允许测试企业名单进入新商业化链路。

### M1.5 自动化验证

- [ ] 订单签名错误、未知企业、未知版本、金额不符测试。
- [ ] 同订单100次并发回调只入账一次。
- [ ] 余额1时100次并发展示最多成功扣量一次，余额不为负。
- [ ] A企业不能读取或消费B企业额度。
- [ ] 30分钟去重、30分钟后再次计量、预览免扣量测试。
- [ ] 事务在访问记录或账本写入失败时整体回滚。
- [ ] 现有名片展示、分享、点赞和访问统计回归测试。

### M1 验收演示

```text
测试企业购买起步展示包
→ 企业微信回调被验证并只处理一次
→ 管理端显示1年有效、10000次
→ 外部访客首次打开扣1次
→ 30分钟内重复打开不扣
→ 第二访客打开后剩余9998次
→ 账本、访问统计和租户隔离一致
```

## 6. M2：完整产品与运营闭环

### M2.1 版本生命周期

- [ ] 基础体验版首次授权赠送500次、7天先到为准，重新安装不重复赠送。
- [ ] 三个收费版本全部接通真实外部版本 ID。
- [ ] 续购创建新额度批次，优先消费最早到期批次。
- [ ] 到期任务将剩余额度标记失效并写不可变账本。
- [ ] 按 M0 证据实现退款/撤销反向流水；复杂冲突进入人工审核。
- [ ] 只有平台明确支持后才开放升级和差价逻辑。

### M2.2 企业管理端

- [ ] 版本和额度总览、7/30天趋势、成员/名片用量排行。
- [ ] 订单列表和稳定状态解释，不展示原始回调载荷。
- [ ] 购买/续购入口和成功回跳。
- [ ] 20%、10%、0和到期提醒状态。

### M2.3 平台管理端

- [ ] 异常订单、退款待处理、版本映射、通知失败队列。
- [ ] 额度人工调整需要原因、二次确认和审计记录。
- [ ] 导出仅包含必要账务字段并记录操作者。

### M2.4 通知

- [ ] Outbox worker、指数退避、死信和人工重放。
- [ ] 购买成功、额度20%/10%/0、到期30/7/1天模板。
- [ ] 企业取消授权后停止发送；无有效管理员时记录可观测失败。
- [ ] 通知失败不影响订单入账和展示扣量。

### M2.5 审核体验清理

- [ ] 删除生产环境“读取失败，已展示演示名片”的降级行为。
- [ ] 隐藏纸质名片识别、名片交换等“即将上线”入口。
- [ ] 用全新测试企业验证授权后自动创建企业、成员和默认名片。
- [ ] 企业微信桌面端保持“无额外适配”的如实选择，并完成基础可用性测试。

## 7. M3：生产上线门槛

### M3.1 性能和可靠性

- [ ] 对同企业热点名片进行并发压测，验证锁等待、吞吐和数据库连接池。
- [ ] 对 `display_dedupe_states` 制定31天清理任务，验证不影响30分钟窗口。
- [ ] 验证订单 outbox、通知 worker 和到期任务在进程重启后可恢复。
- [ ] 建立数据库备份、恢复演练和商业化功能独立关闭开关。

### M3.2 监控告警

- [ ] 订单回调成功率、应用耗时、manual review 数量。
- [ ] 扣量成功/去重/预览/耗尽计数和负余额尝试。
- [ ] 通知积压、失败和定时任务心跳。
- [ ] 支付成功超过5分钟未入账、同订单字段冲突、额度账实不符立即告警。

### M3.3 合规和材料同步

- [ ] `docs/04-launch/05_功能及定价表.md` 改为体验版加三个收费版本。
- [ ] `03_第三方服务协议_正式版.md`、`08_用户服务协议_正式版.md` 和小程序内协议统一收费、额度、有效期、退款和到期规则。
- [ ] 隐私政策核对访问去重、匿名标识、通知及订单信息处理说明。
- [ ] 更新上线审核清单，移除“完全免费、不创建收费版本”的旧结论。
- [ ] 客服形成订单查询、到账失败、额度争议、退款和申诉 SOP。
- [ ] 配置发票、税率、结算账户和依法需要的账务留存。

### M3.4 灰度和回滚

- [ ] 先对白名单测试企业启用，再开放少量真实企业。
- [ ] 旧企业切换前生成基线订阅/体验权益，不允许直接进入额度耗尽。
- [ ] 回滚代码时保持订单回调可接收和持久化，禁止丢事件。
- [ ] 紧急情况下可停止扣量并允许展示，但任何临时免费放行必须有开始/结束时间、审批和审计记录。

## 8. 任务依赖与建议顺序

```text
M0真实订单与通知证据
  → 数据模型冻结
  → v1_10迁移与RLS
  → 订单幂等入账
  → 展示准入与原子扣量
  → 小程序公开页切换
  → 企业/平台管理端
  → 通知与定时任务
  → 退款/到期/续购
  → 合规材料统一
  → 压测、灰度、上线
```

订单协议未验证不阻塞独立完成展示扣量的本地领域模型和并发测试，但阻塞真实订单字段冻结、生产售卖和退款自动化。

## 10. Readiness Gate

- Done definition：已在第1节定义。
- Validation method：平台实测证据 + 数据库迁移验证 + 自动化并发/隔离测试 + 测试企业端到端录像。
- Fail condition：订单身份、签名、版本或金额无法可靠验证，或计费口径不获平台允许。
- P0 unknowns：真实订单/退款回调协议、版本字段、通知接口和平台结算规则。
- P1 before-module：公开名片读取必须纳入展示准入；商业表必须有RLS和不可变账本；正式材料必须移除免费版冲突。
- **Verdict：Ready for M0。可以立即开始平台协议取证和本地领域模型设计；生产订单模块与正式收费必须等待 M0 通过。**

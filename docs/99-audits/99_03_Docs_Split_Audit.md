# Audit #03 — 文档拆分一致性审计

日期：2026-07-01
范围：docs/ moread 化重构 + 抽取的 6 份指引（01_01 Wecom、00_02 DB Schema、01_02 API、01_03 小程序、01_04 后台、03_01 PIPL）
目标：验证拆分后**交叉引用可达、单一事实源不被破坏、无内容漂移/矛盾、覆盖无缺口**。
方法：脚本校验全部 `.md` 相对链接；逐份对照主文档 §ref；人工核对重复/矛盾。

## 结论

拆分整体健康：**所有相对链接可达（脚本验证）**；各 spec 以「引用主文档 + 补执行细节」方式落地，未见 DDL / 接口的大段复制，单一事实源基本守住。存在 2 处 P2 漂移需收口，1 处 P3 覆盖待补。

## findings

### P2-1：API 路径版本漂移（master §14 vs 01_02_Api_Spec）
- 现状：`01_02_Api_Spec.md` 用 `/api/v1/...`（依 §28.1 决策），但主文档 §14 示例仍是 `/api/auth/wx-login` 等**无 v1** 路径。
- 风险：两处路径不一致，开发按哪份走会歧义；且 §14 仍列全量路径 = 双维护，易漂移。
- 建议：主文档 §14 只保留**接口分组 + 指针**，删除具体路径示例，令 `01_02_Api_Spec` 成为路径唯一事实源。

### P2-2：`action_type` 取值目录分散
- 现状：枚举目录在 `00_02_Database_Schema.md` §4（含 `copy_phone` 等），主文档 §14.3 示例另列一小串（`save_phone/call_phone/copy_email/add_wecom`），小程序指引又引 schema §4。
- 风险：三处口径可能各自增改。
- 建议：以 schema §4 为唯一目录，主文档 §14.3 与 API spec 改为「取值见 00_02 §4」。

### P3-1：88-planning / 02-tasks 未落地
- 现状：里程碑(§19)、Issue 拆分(§21)、启动建议(§25)、路线图/商业化仍只在主文档；`02-tasks/`、`88-planning/` 仅有 `.gitkeep`。
- 判断：可接受（这些偏规划、主文档已自洽），非缺陷。建议进入执行排期时再抽 `02-tasks/02_01_Milestone_Plan.md`，避免提前复制。

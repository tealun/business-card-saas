# 智云名片商业化、展示额度与通知规格

状态：规划基线（待 M0 平台实测）  
制定日期：2026年7月16日  
关联执行计划：[`../02-tasks/02_03_Wecom_Commercialization_Plan.md`](../02-tasks/02_03_Wecom_Commercialization_Plan.md)

## 1. 目标与边界

本规格定义智云名片企业微信第三方应用上线后的商业化闭环：企业购买一年期版本，获得由该企业全部成员名片共享的有效展示额度；系统接收并核验企业微信订单事件，开通版本和额度，在公开名片成功展示时原子扣减，并向企业管理员发送购买、余额、用尽和到期通知。

本阶段不包含：

- 自建微信支付、支付宝或线下收款后自动开通；
- 按员工人数计费；
- 自动续费；
- 额度跨企业转移、兑换现金或人工销售折扣系统；
- 客户联系、会话存档、OA、打卡、审批、发票等未实际接入的企业微信能力；
- 未经企业微信平台确认的升级差价、按比例退款或跨版本余额折算。

## 2. 已确定的产品决策

### 2.1 版本与价格

企业微信后台当前展示的原生模型为“一年期版本 + 固定总价”，不是通用余额充值。产品采用基础体验版加三个收费版本：

| 版本 | 后台名称 | 价格 | 企业共享额度 | 周期 | 自动续费 |
|---|---|---:|---:|---|---|
| trial | 基础体验版 | 0元 | 500次 | 7天，时间或额度先到即结束 | 否 |
| starter | 起步展示包 | 299元 | 10,000次 | 1年（365天） | 否 |
| growth | 成长展示包 | 999元 | 50,000次 | 1年（365天） | 否 |
| enterprise | 企业展示包 | 2,999元 | 200,000次 | 1年（365天） | 否 |

收费版本在企业微信后台统一选择“固定总价”，不启用人数限制、总价限制和购买优惠。平台实际生成的版本 ID 必须保存为外部映射，业务代码不得根据中文版本名判断权益。

### 2.2 功能范围

四个版本提供相同的核心产品功能，主要差异为展示额度和服务优先级：

- 企业微信安装授权和成员身份识别；
- 企业电子名片创建、编辑、展示和分享；
- 企业统一字段、模板和企业资料管理；
- 成员主动授权的企业微信头像和个人二维码同步；
- 名片访问及互动基础统计；
- 企业全部成员共享版本内展示额度。

不得在价格表中承诺尚未上线的纸质名片识别、名片交换、AI 生成、CRM、客户群、会话存档或广告投放分析。

### 2.3 有效展示定义

一次“有效展示”必须同时满足：

1. 访问的是处于可展示状态的真实企业成员公开名片；
2. 服务端已确认企业版本有效且存在可用额度；
3. 服务端成功返回公开名片主要内容；
4. 访问未被识别为企业成员本人预览、企业管理员预览、健康检查、爬虫、攻击或明显异常流量；
5. 同一访问者在30分钟内重复打开同一张名片时，只计量一次。

以下行为不额外扣量：点赞、分享、复制、拨打、保存联系人、读取同一页面的静态资源和接口自动重试。打开不同成员的名片分别计量。

“同一访问者”优先使用服务端认证的访客账号；未登录访客使用服务端签名的匿名标识。不得接受客户端自行声明“预览”“不扣费”或直接上送可信企业、用户身份。

### 2.4 额度生命周期

- 企业首次成功授权时，可开通一次基础体验权益；同一企业取消授权后重新安装，不重复赠送。
- 购买或续购成功后创建不可变的额度批次，每个批次记录来源订单、总量、剩余量、生效和到期时间。
- 多个有效批次并存时，优先扣减最早到期的批次。
- 批次到期后，未使用额度失效，不结转、不转让、不兑换现金。
- 订单退款、撤销或冲正时不得直接删除历史流水；应创建反向调整记录。已消费额度超过可撤销额度时进入人工处理，不允许余额变为负数。
- 当前版本不默认承诺付费额度耗尽后的宽限期。72小时或500次宽限方案列为 M2 可选项，启用前必须同步价格说明、协议、页面和通知文案。

### 2.5 额度耗尽体验

- 企业成员和管理员仍可登录、编辑、预览、查看余额和购买版本。
- 未获得服务端预览权限的外部访客不再获取完整名片数据，页面展示通用提示：“当前企业的名片展示服务暂时不可用，请稍后再试或联系该企业管理员。”
- 外部页面不得展示“企业没钱”“余额不足”或具体购买价格。
- 企业管理员后台展示当前版本、有效期、总额度、已用、剩余、最近消费明细和购买入口。

## 3. 角色与权限

| 角色 | 允许操作 | 禁止操作 |
|---|---|---|
| 外部访客 | 在额度有效时查看公开名片并产生有效展示 | 查询企业余额、订单或消费明细 |
| 企业成员 | 编辑和预览本人名片、查看本人访问统计 | 绕过扣量公开展示、查看企业订单（非管理员） |
| 企业管理员 | 查看本企业版本、余额、订单、消费汇总和通知状态 | 查看其他企业数据、修改平台订单金额或直接增加额度 |
| 平台管理员 | 查看全局版本映射、异常订单、人工调整和通知失败 | 无审计记录地修改额度或删除账务流水 |
| 企业微信回调 | 经签名、解密和幂等校验后变更订单及权益 | 通过普通公开接口直接写入权益 |

所有企业订单、额度、消费和通知表必须按 `tenant_id` 隔离。平台版本目录可以是全局表，但不得包含企业个人信息。

## 4. 系统架构

建议新增后端领域模块：

```text
backend/src/commercialization/
  plan-catalog.*          # 内部套餐与企业微信版本 ID 映射
  order-callback.*        # 订单事件验签、解密、幂等和状态机
  subscription.*          # 企业版本生命周期
  quota-account.*         # 额度批次、原子扣减和调整
  display-access.*        # 公开展示准入、去重和扣量
  notification-outbox.*  # 通知生成、重试和投递结果
```

与现有模块的边界：

- `wecom/` 继续负责企业微信通用加密、令牌和回调入口；商业订单事件解密后交给 `commercialization/`。
- `public-card/` 不自行维护价格和版本，仅调用 `DisplayAccessService` 获取展示准入结果。
- `admin-*` 只通过商业化服务查询本企业权益和平台异常，不直接改账。
- `metrics/` 增加订单、扣量和通知指标，不把订单号、企业名称或个人信息写入标签。

### 4.1 公开展示主链路

现有公开名片“先读取完整名片、再创建访问记录”的顺序不适合付费准入，因为额度不足时数据已经返回。目标链路为：

```text
访客打开公开名片
  → POST /api/v1/public/cards/{public_id}/display-session
  → 解析公开目录并校验名片/租户状态
  → 识别可信预览或计算匿名访客去重键
  → 数据库事务内锁定去重状态与额度批次
  → 判断30分钟内是否已计量
  → 必要时原子扣减1次并写入额度流水
  → 创建 card_visit 并签发 visit_token
  → 返回公开名片内容、访问令牌和计量结果
```

公开访客不得继续通过旧接口绕过 `display-session` 直接取得完整名片。员工或管理员预览使用受认证的独立路径，服务端校验所属企业及权限后免扣量。

## 5. 数据模型草案

数据库事实源仍为 `database/schema.sql`。实施时使用下一条前向迁移 `database/migrations/migrate_v1_10.sql`，并同步 [`../00-core/00_02_Database_Schema.md`](../00-core/00_02_Database_Schema.md)。

### 5.1 `commercial_plans`

全局版本目录：

- `id`, `code`（trial/starter/growth/enterprise，唯一）；
- `name`, `price_fen`, `duration_days`, `display_quota`；
- `status`, `created_at`, `updated_at`。

### 5.2 `wecom_plan_mappings`

企业微信外部版本映射：

- `id`, `plan_id`；
- `provider_app_id` 或平台要求的应用标识；
- `external_version_id`（唯一）；
- `environment`（test/production）；
- `status`, `created_at`, `updated_at`。

外部字段名和长度必须以 M0 捕获的真实接口响应为准。

### 5.3 `tenant_subscriptions`

- `id`, `tenant_id`, `plan_id`；
- `status`（trial/active/expired/cancelled/refund_pending）；
- `source_order_id`, `starts_at`, `ends_at`；
- `created_at`, `updated_at`；
- 同一企业允许保留历史记录；当前有效订阅通过部分唯一索引或事务不变量控制。

### 5.4 `billing_orders`

- `id`, `tenant_id`, `plan_id`, `subscription_id`；
- `provider`, `external_order_id`, `external_version_id`；
- `event_type`, `status`, `amount_fen`, `currency`；
- `purchased_at`, `effective_at`, `expires_at`；
- `payload_encrypted`, `received_at`, `processed_at`, `last_error`；
- `external_order_id` 唯一；原始敏感载荷加密存储，日志只记录内部 ID。

### 5.5 `quota_grants`

- `id`, `tenant_id`, `subscription_id`, `billing_order_id`；
- `grant_type`（trial/purchase/renewal/adjustment/grace）；
- `total_units`, `remaining_units`；
- `starts_at`, `expires_at`, `status`；
- `created_at`, `updated_at`；
- `remaining_units >= 0`，扣量按最早到期批次处理。

### 5.6 `quota_ledger`

不可变账本：

- `id`, `tenant_id`, `grant_id`；
- `entry_type`（grant/consume/expire/refund/adjustment/reversal）；
- `units`（有符号整数）；
- `idempotency_key`（唯一）；
- `card_id`, `visit_id`, `external_order_id`（按类型可空）；
- `reason_code`, `actor_type`, `actor_id`, `created_at`。

任何人工调整必须有操作者、原因和唯一幂等键，不允许 UPDATE/DELETE 历史账本。

### 5.7 `display_dedupe_states`

- `tenant_id`, `card_id`, `visitor_key_hash` 组成唯一键；
- `last_counted_at`, `last_visit_id`, `updated_at`。

事务先 `INSERT ... ON CONFLICT DO NOTHING`，再 `SELECT ... FOR UPDATE`；若 `last_counted_at > now() - interval '30 minutes'`，创建访问记录但不扣量。`visitor_key_hash` 使用服务端密钥 HMAC，不保存可逆身份。

### 5.8 `notification_outbox` 与 `notification_deliveries`

Outbox 记录业务通知意图，Delivery 记录每次投递：

- 企业、模板代码、接收管理员、去重键、变量 JSON；
- 状态、尝试次数、下次重试时间、平台消息 ID、错误码；
- 发送成功、永久失败和重试耗尽时间；
- 通知内容不得包含完整订单载荷、token、secret 或个人二维码。

### 5.9 RLS、权限和留存

- 商业企业表启用 RLS；无 `app.tenant_id` 默认拒绝。
- 平台管理员跨租户读取必须经过独立平台鉴权，不以关闭 RLS 代替授权。
- 后端应用角色可在显式事务内设置租户上下文；客户端不直连商业表。
- 账务和订单记录按财税、争议和平台规则确定期限；在 M0 未取得规则前不得承诺自动短期删除。
- 展示去重状态可在最后计量后31天清理；通知投递明细建议保留180天，最终期限需合规确认。

## 6. API 契约草案

正式接口需同步 [`01_02_Api_Spec.md`](01_02_Api_Spec.md)。

| 方法与路径 | 鉴权 | 用途 |
|---|---|---|
| `POST /api/v1/public/cards/{public_id}/display-session` | 公开；匿名标识必须服务端签名 | 展示准入、去重、扣量、访问记录和名片返回 |
| `GET /api/v1/admin/commercial/subscription` | tenant admin | 当前版本、有效期和额度摘要 |
| `GET /api/v1/admin/commercial/orders` | tenant admin | 本企业订单列表，不返回加密载荷 |
| `GET /api/v1/admin/commercial/usage` | tenant admin | 按日期、成员、名片汇总展示用量 |
| `GET /api/v1/platform/commercial/orders/exceptions` | platform admin | 异常订单和人工处理队列 |
| `POST /api/v1/platform/commercial/quota-adjustments` | platform admin + 强审计 | 经审批原因进行额度调整 |

订单回调优先复用企业微信已有加密回调 URL；若官方要求独立 URL，再新增专用入口。不得在未验证官方协议前固定回调路径、字段或签名方法。

关键响应状态：

- `active`：允许展示，可能扣量；
- `deduplicated`：允许展示，本次不扣量；
- `preview`：可信员工/管理员预览，不扣量；
- `trial_expired`：体验期结束；
- `quota_exhausted`：额度用尽；
- `subscription_expired`：版本到期；
- `tenant_cancelled` / `card_unavailable`：沿用现有状态语义。

客户端只根据稳定错误码展示文案，不解析后端英文异常字符串。

## 7. 订单状态机与幂等

```text
received → verified → applied
                   ↘ retryable_failed → applied
                   ↘ manual_review
received → invalid_signature（拒绝且告警）
applied → refunded / reversed（反向流水）
```

- 回调必须验证签名、时间戳、接收方标识和加密载荷，沿用现有企业微信安全边界。
- 使用官方事件唯一 ID；若没有，使用稳定业务字段生成复合幂等键。不得仅以回调到达时间去重。
- 同一订单重复、乱序和并发回调只能产生一次有效权益变更。
- 先持久化验证后的事件和幂等状态，再处理权益；异常可重放，不能要求企业再次付款。
- HTTP 回应遵循官方时限和确认格式；业务异步处理采用 outbox/retry，不在回调线程发送通知。

## 8. 应用通知模板

规划模板：

| 模板代码 | 标题 | 触发条件 | 接收人 |
|---|---|---|---|

| `quota_low_10` | 名片展示额度预警 | 可用额度首次低于10% | 企业有效管理员 |
| `quota_exhausted` | 名片展示额度已用尽 | 可用额度从正数变为0 | 企业有效管理员 |
| `subscription_expiry_30d` | 智云名片版本到期提醒 | 距到期30天 | 企业有效管理员 |
| `subscription_expiry_7d` | 智云名片版本到期提醒 | 距到期7天 | 企业有效管理员 |
| `subscription_expiry_1d` | 智云名片版本到期提醒 | 距到期1天 | 企业有效管理员 |

同一企业、订阅和阈值只生成一次通知。通知失败不得回滚订单或额度交易；采用指数退避重试，超过上限进入告警和人工补发队列。

模板 ID 不能硬编码在源码。测试和生产映射分别配置，启动时校验必需模板是否完整。

## 9. 管理端和小程序需求

### 9.1 企业管理端

- 当前版本、价格、起止时间和状态；
- 总额度、已用、剩余和使用百分比；
- 近7/30天用量趋势、按成员/名片的汇总；
- 订单列表、订单状态和企业微信购买入口；
- 低余额、到期和异常状态提示；
- 不向普通企业管理员提供改余额按钮。

### 9.2 平台管理端

- 企业微信版本 ID 与内部套餐映射；
- 异常、重复、退款待处理订单；
- 额度账本查询和带原因的人工调整；
- 通知失败重放；
- 所有敏感操作写审计日志。

### 9.3 小程序和公开页

- 员工端显示企业版本和余额提示，但不得让普通员工查看订单金额和购买人信息；
- 外部访客在额度用尽或版本到期时显示通用不可用态；
- 企业管理员可从提示进入企业微信平台购买入口；
- 审核版本隐藏“即将上线”和演示数据入口，生产环境读取失败不得降级成演示名片。

## 10. 安全、合规与运营要求

- 金额以整数分存储，额度以整数次存储，禁止浮点计算。
- 订单、权益、扣量、退款和调整均需可追溯；不在日志输出回调密文、token、手机号、二维码或完整订单载荷。
- 扣量操作必须在单个数据库事务内完成，任意异常整体回滚。
- 企业取消授权后停止刷新企业 token 和发送应用消息，但保留依法需要的账务记录；后续处置依平台规则和协议执行。
- 隐私政策应覆盖匿名标识、访问时间和名片互动统计；价格表、服务协议、小程序内协议和购买页必须保持版本、额度、有效期、退款口径一致。
- 客服需能查询内部订单 ID、订阅状态、额度流水和通知结果，但默认看不到不必要的个人信息。

## 11. 可观测性

新增无高基数标签的指标：

- `wecom_order_callback_total{event,status}`；
- `commercial_order_apply_duration_seconds`；
- `quota_consumption_total{result}`；
- `quota_exhausted_tenants`；
- `commercial_notification_total{template,status}`；
- `commercial_outbox_backlog`。

告警条件至少包括：有效签名回调连续失败、同订单金额/版本冲突、额度出现负数尝试、订单已支付但权益未入账、通知积压和定时任务停止。

## 12. 决策、假设与待验证台账

### Decisions

- D1：采用一年固定总价版本，不在本阶段自建通用充值支付。
- D2：额度归企业共享，不按成员收费。
- D3：展示准入与扣量在服务端数据库事务内完成。
- D4：使用不可变额度账本，退款和调整通过反向流水完成。
- D5：外部访客不能在额度不足时取得完整名片；可信预览不扣量。

### Assumptions

- A1：企业微信订单事件能够稳定关联授权企业和后台版本 ID。
- A2：企业微信允许一年期固定总价版本附带业务侧使用额度。
- A3：企业微信应用通知可以向授权企业的有效管理员发送上述模板消息。
- A4：一个企业在任一时刻的付费权益可归一为一个当前订阅和若干未过期额度批次。

### To Verify

| ID | M0 所需证据 | 负责人 | 失败时处理 |
|---|---|---|---|
| V1 | 企业微信官方订单/版本接口文档、申请权限截图和测试环境真实请求响应 | 平台接入 | 不开发订单写入，仅保留手工试点沙箱 |
| V2 | 购买、续购、升级、退款、取消、到期的真实回调样本及签名规则 | 后端 | 阻塞状态机和生产售卖 |
| V3 | 外部版本 ID、企业标识、订单 ID、金额和有效期字段的真实类型/长度 | 后端/数据库 | 不冻结表字段和唯一约束 |
| V4 | 版本购买入口、购买完成回跳及管理员接收通知的真机录像 | 小程序/运营 | 不开放购买入口和消息承诺 |
| V5 | 平台对提前续购、升级差价、退款后已消费额度的规则说明 | 产品/运营 | 首版禁用升级/退款自动化，进入人工审核 |
| V6 | 应用通知模板 ID、发送接口、频率限制和错误码实测 | 后端 | 首版只在管理端展示提醒，不发企微通知 |
| V7 | 价格表、服务协议、隐私政策和发票/税务资料审核结果 | 运营/合规 | 阻塞收费版本对外发布 |

任何影响身份、订单签名、金额、权益或退款的外部事实未验证前，商业化模块只能进入 M0，不得宣称可生产收费。
